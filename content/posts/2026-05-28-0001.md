# kimi-cli 安装记录

## 环境信息

- **操作系统**: Ubuntu (wsl2)
- **Python 版本**: 3.12.2（通过 pyenv 安装）
- **包管理器**: uv（通过 pip 安装在虚拟环境中）

## 安装步骤

### 1. 安装 Python 3.12.2

使用 pyenv 安装 Python 3.12.2：

```bash
pyenv install 3.12.2
```

### 2. 创建虚拟环境

```bash
python -m venv ./venv
source venv/bin/activate  # Windows 使用: venv\Scripts\activate
```

### 3. 安装 uv

```bash
pip install uv
```

### 4. 安装 kimi-cli

使用 uv 的工具安装功能，指定 Python 3.13 版本：

```bash
uv tool install --python 3.13 kimi-cli
```

> **说明**: 即使本地 Python 版本是 3.12.2，也可以指定 `--python 3.13`。uv 会自动下载 Python 3.13 并为 kimi-cli 创建独立的隔离环境。

## 常见问题解答

### Q1: 我的 Python 版本是 3.12.2，`--python` 参数是不是要填我自己的版本？

**A**: 不需要。`--python 3.13` 是让 uv 使用 Python 3.13 来安装和运行 kimi-cli。由于本地是 3.12.2，有几种选择：

- **推荐**: 去掉 `--python` 参数，uv 会自动使用现有 Python 3.12.2：
  ```bash
  uv tool install kimi-cli
  ```
- **指定当前版本**:
  ```bash
  uv tool install --python 3.12 kimi-cli
  ```
- **使用官方推荐方式**（如本文档所示）: uv 会自动下载并管理 Python 3.13

### Q2: 我用 pyenv 安装了 Python 3.12.2 并创建了虚拟环境，在虚拟环境里又安装了 uv，现在用 uv 安装 kimi-cli 会不会有问题？是否相当于嵌套虚拟环境？

**A**: 完全没问题，这是 uv 的正确用法。

**关键概念**:
- `uv tool install` 是 uv 的**工具管理**功能
- 它会在 uv 自己的隔离目录（如 `~/.local/share/uv/tools`）中安装 kimi-cli
- 为其创建**独立环境**，不会安装到当前的 pyenv 虚拟环境中
- 与 pyenv 完全不冲突

**实际过程**:
1. uv 检测到本地没有 Python 3.13 -> **自动下载** Python 3.13
2. 用下载的 3.13 创建一个**隔离环境**专门给 kimi-cli 使用
3. pyenv 的虚拟环境（3.12.2）**完全不受影响**

**结论**: 不是在"嵌套虚拟环境"，而是让 uv 独立管理命令行工具。这是 uv 的设计优势——每个工具都有自己的隔离环境，互不干扰。

## 验证安装

安装完成后，验证 kimi-cli 是否可用：

```bash
kimi --version
```

## 补充

- 使用kimi-cli的时候不需要进入之前创建的venv虚拟环境，因为直接调用了.local/share/uv/python/cpython-3.13-linux-x86_64-gnu

## 备注

- uv 的 `tool install` 功能类似于 `pipx`，用于安装隔离的 Python 命令行工具
- 每个通过 uv 安装的工具都有自己的虚拟环境，不会与系统 Python 或项目虚拟环境冲突
